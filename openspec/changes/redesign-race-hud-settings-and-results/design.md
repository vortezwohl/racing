## Context

当前比赛内 UI 是混合式实现：

- 比赛主画面使用 `Three.js` 渲染到 `#game` canvas。
- HUD、倒计时和结算页仍然依赖 `#dashboard`、`#finish-screen` 等 DOM 节点。
- `Track.ts` 直接写计时文本，`Vehicle.ts` 直接写圈数文本，`GameScene.ts` 直接控制结算页显隐与内容。
- 比赛页目前没有统一的点击式 UI 命中层，只有菜单页拥有射线点击逻辑，比赛页主要依赖键盘和触屏摇杆。

这带来几个直接问题：

- HUD 的信息来源分散，难以整体迁移到 Canvas。
- 设置按钮、弹窗和结算操作如果继续追加 DOM，会让比赛内 UI 进一步碎片化。
- 结算页想保留赛道背景并持续更新 NPC 冲线结果，就不能再沿用“黑场 + 一次性静态结算 DOM”的旧思路。
- 重开入口一旦同时来自设置弹窗和结算面板，就必须与现有单页比赛会话重建、随机发车位逻辑严格对齐。

约束条件：

- 保持现有 `webpack + TypeScript + Three.js` 架构，不引入新的前端框架。
- 保持菜单页与比赛页现有单页壳体结构。
- 尽量减少对核心物理、AI、赛道渲染和菜单流程的侵入。
- 文件输出与注释需要继续遵守中文约定，但 OpenSpec 规范标题和 requirement 结构需保持可解析格式。

## Goals / Non-Goals

**Goals:**

- 为比赛页引入独立的 Canvas HUD 层，统一渲染圈数、名次、计时、时速、倒计时、齿轮按钮、设置弹窗和结算面板。
- 收口比赛内 UI 状态来源，避免 `Track`、`Vehicle`、`GameScene` 继续分别直写不同 DOM 节点。
- 保持比赛不会因打开设置弹窗而暂停，并支持在弹窗中退出到菜单或重开比赛。
- 用半透明叠加结果面板替代当前黑场式结算页，同时允许结算状态继续接收 NPC 冲线结果更新。
- 让所有 `RETRY / 重新开始比赛` 路径都走同一套新会话创建逻辑，并重新随机化发车位。

**Non-Goals:**

- 不重写菜单页交互模型。
- 不调整车辆物理参数、NPC 驾驶策略或赛道几何。
- 不在本次设计中引入新的资源打包方式、Web 字体加载系统或第三方 HUD 库。
- 不追求把所有比赛内 DOM 一次性彻底删除；只收口当前任务必须覆盖的 HUD、设置和结算 UI。

## Decisions

### 1. 为比赛页新增一层专用 HUD Canvas，而不是把文字和按钮塞回 Three.js 场景

比赛内 UI 将使用独立的 2D Canvas 叠加层渲染，位置在 `#game` 之上、`marker-layer` 与 `joystick` 的适当层级之间或之上。该层负责所有用户明确要求的界面元素：

- 左上：圈数与名次
- 中上：计时与倒计时
- 右上：时速与齿轮按钮
- 中央：设置弹窗
- 赛后：半透明结算面板与 `BACK` / `RETRY`

采用这条路线的原因：

- 用户明确要求这些元素“都用 canvas 绘制”。
- 2D Canvas 更适合排版文本、做霓虹描边、控制透明玻璃面板和精细点击区域。
- 相比把 HUD 做成 Three.js sprite / mesh，这种方式不需要把相机、景深、Bloom 和 UI 混到一起，维护风险更低。

备选方案：

- 继续沿用 DOM HUD，只改 CSS。放弃，因为这只能换皮，不能满足“都用 canvas 绘制”。
- 把 HUD 做进 Three.js 场景。放弃，因为文本布局、响应式、点击命中和结算面板都会更复杂。

### 2. 引入 `GameScene` 所拥有的比赛 UI 状态对象，逐步替代分散的 DOM 直写

比赛内 UI 将由 `GameScene` 维护单一状态快照，例如：

- 当前圈数、总圈数
- 实时名次、总参赛车数
- 当前速度值与显示用 `km/h`
- 倒计时文本
- 比赛计时文本
- 设置弹窗开关
- 结算面板开关
- 玩家单圈 split、平均时速、完赛时间
- 排行榜条目、已完赛 NPC 状态
- 当前可点击热区

`Track.ts` 与 `Vehicle.ts` 不再承担 HUD 文本更新职责。它们继续维护底层比赛数据，如 `laps`、`velocity`、`elapsedTime`、`checkpoint`，由 `GameScene` 每帧汇总后交给 HUD Canvas 渲染。

采用这条路线的原因：

- 当前 `Track.ts` 和 `Vehicle.ts` 直接写 DOM，会阻碍 Canvas HUD 收口。
- 比赛 HUD、设置弹窗和结算面板共享大量状态，统一持有更容易保证一致性。
- 这也更符合已有单页壳体中“由场景持有会话状态”的方向。

备选方案：

- 继续允许 `Track`、`Vehicle` 更新 UI，只把最终文本再抄到 Canvas。放弃，因为这会保留双份状态源，后续维护更容易错乱。

### 3. 比赛时间与结算数据分离：显示时间可冻结，赛会时钟仍可继续用于 NPC 完赛记录

当前 `Track.elapsedTime` 在玩家完赛后会停止推进，现状足以显示玩家成绩，但不足以支持“玩家已完赛后，NPC 若随后冲线，也继续更新结果面板”这一需求。

本次设计建议把“用于结果统计的赛会时钟”与“HUD 当前显示时间”拆开：

- 赛中 HUD 计时显示实时推进。
- 玩家完赛时记录其完赛时间、平均时速与单圈 split。
- 结果面板显示玩家已冻结的成绩。
- 赛会时钟在结果收集阶段继续推进，以便记录 NPC 后续完赛时间和更新排行榜。

这不要求用户继续看到变化中的中心计时；中心计时可以在完赛后冻结为玩家成绩，但后台结果采集仍继续。

采用这条路线的原因：

- 能满足用户要求的“NPC 冲线后也更新到面板上”。
- 避免直接继续滚动玩家的完赛时间，保证主成绩稳定。

备选方案：

- 玩家完赛后彻底停止整场逻辑。放弃，因为无法继续收集 NPC 冲线信息。
- 玩家完赛后继续滚动中心计时。放弃，因为会弱化玩家成绩的完成感。

### 4. 统一将重开操作解释为“创建新的比赛会话”，而不是在旧会话上局部 reset

来自两处的重开入口：

- 设置弹窗中的“重新开始比赛”
- 结算面板中的 `RETRY`

都将回到同一条壳体层级的 race 重进路径，由 `AppShell` 创建全新的 `GameScene` 实例，而不是在现有实例上做局部清空。

采用这条路线的原因：

- 现有 `app.ts` 已经在每次进入比赛时新建 `GameScene`，与现有会话边界自然一致。
- 随机发车位逻辑已经在 `GameScene.render()` 初始化阶段完成，重建场景即可天然重新洗牌。
- 相比手工 reset 各种倒计时、成绩、NPC 状态、点击区域和 UI 开关，直接新建会话更不容易遗漏。

备选方案：

- 保留旧 `GameScene` 并手动 reset。放弃，因为状态项会明显增多，容易留下脏状态。

### 5. 比赛内设置按钮与结果操作统一采用 HUD Canvas 热区命中，不引入新的 DOM 按钮

HUD Canvas 渲染器在每帧或每次重绘后，都会同步维护一组逻辑热区矩形：

- 齿轮按钮
- 设置弹窗内的“重新开始比赛”
- 设置弹窗内的“退出到菜单”
- 结算面板的 `BACK`
- 结算面板的 `RETRY`

比赛页只需要新增一套轻量 pointer/click 监听，将屏幕坐标映射到 HUD Canvas 热区进行命中测试。

采用这条路线的原因：

- 这能保持“看见什么，就点什么”的 Canvas UI 一致性。
- 比赛页当前没有点击 UI 体系，新增一层纯逻辑热区比新增多套 DOM 更简单。
- 后续若继续扩展赛中按钮、悬停样式或移动端适配，也能沿用这套结构。

备选方案：

- 只用 DOM 透明按钮覆盖在 Canvas 上。放弃，因为会破坏“用 canvas 绘制”的一致性，也会带来额外定位同步问题。

### 6. 时速表显示为经校准的游戏内 `km/h`，不把当前物理速度伪装成严格真实世界单位

当前车辆速度模型是游戏化参数驱动，并不是已经与真实世界距离/秒严格标定的物理系统。因此本次设计把右上角速度表定义为“稳定、直观、可比较的游戏内 `km/h` 显示”，而不是承诺它是严格现实意义上的物理时速。

实现层面可以采用：

- 依据当前速度向量长度映射到固定上限的显示值
- 或依据 `getSpeedRatio()` 与最大显示速度换算

采用这条路线的原因：

- 用户明确要的是时速表和质感，而不是物理仿真校准工具。
- 这样能避免未来为保持数值“像真实 km/h”而牵一发动全身地改动物理系统。

备选方案：

- 把当前速度彻底重标定成真实物理单位。放弃，因为超出本次 UI 重构范围。

## Risks / Trade-offs

- [HUD 状态从分散 DOM 写入迁移到场景收口时容易漏改] → 先明确列出 `Track.ts`、`Vehicle.ts`、`GameScene.ts` 的 UI 写入点，逐个替换为状态驱动，不保留双写。
- [结果面板需要在玩家完赛后继续更新 NPC 结果，可能暴露旧计时模型不足] → 在设计中明确引入赛会结果时钟与玩家显示时间分离，避免直接复用已冻结的 HUD 时间。
- [Canvas 点击热区在不同分辨率下可能偏移] → 让热区基于 HUD Canvas 的实际像素尺寸和当前布局函数生成，并把点击坐标统一转换到同一坐标系。
- [Tron 风格霓虹字体如果过亮，可能和 Bloom 叠加后发虚] → 在 HUD Canvas 内自行控制描边、发光和透明度，不依赖 3D Bloom 来制造文字发光。
- [继续保留部分旧 DOM 容器可能造成层级冲突] → 本次明确只保留当前任务必须继续存在的 `marker-layer`、`joystick` 与全局 `curtain`，并重新梳理 z-index。
- [不同重开入口若绕过统一会话创建逻辑，可能破坏随机发车位要求] → 让所有重开路径最终都回到 `AppShell.navigateToRace(...)` 或等价的新会话入口。

## Migration Plan

1. 在比赛页壳体中引入独立 HUD Canvas 容器和必要的层级调整，但先不移除旧 HUD DOM。
2. 在 `GameScene` 中建立统一比赛 UI 状态与 HUD 绘制器，先完成只读绘制，再逐步替换旧 HUD DOM 文本更新。
3. 将比赛圈数、计时、倒计时、时速和名次全部切换到 Canvas HUD。
4. 接入齿轮按钮、设置弹窗和点击热区命中逻辑，并将“回菜单 / 重开比赛”接到壳体层的新会话入口。
5. 用新的半透明结果面板替换旧 `finish-screen` 视觉，并接入平均时速、单圈成绩、排行榜和 NPC 后续冲线更新。
6. 收口旧 `finish-screen` 和旧 `dashboard` 的显示逻辑，保留仅为兼容当前会话所必需的最小 DOM。
7. 回归验证比赛开始、赛中设置、退出菜单、赛后结算、`BACK`、`RETRY`、移动端摇杆层级与随机发车位重建。

回滚策略：

- 若新 HUD Canvas 接入后出现大面积交互或层级回归，可先保留新的比赛会话入口和结果数据结构，单独回退到旧 DOM HUD/结算页。
- 由于变更主要集中在比赛内 UI 和 `GameScene` 会话层，可按“Canvas HUD”“设置弹窗”“结果面板”三段独立回退。

## Open Questions

- 结算时是否允许玩家继续有限度控制车辆，还是只保留赛道背景与 NPC 更新而屏蔽玩家操作？
- 设置弹窗关闭方式是否仅支持再次点击齿轮/点击面板外区域，还是需要额外的 `CLOSE` 文本操作？
- 结算排行榜是否显示所有 5 名参赛者，即使有 NPC 尚未完赛，也要以 `DNF / RACING` 之类状态占位，还是只显示已产生结果的条目？
