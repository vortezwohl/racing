## Context

当前比赛尾流的核心渲染结构集中在 [GameScene.ts](D:/Projects/TypeScriptProjects/racing/src/scenes/GameScene.ts) 的 `trailShell / trailFin / trailCore` 三层 mesh 与 `glowSprites` 上。虽然现在已经有速度驱动的长度增长、有效长度裁剪和 draft 判定一致性，但视觉主体仍依赖 ribbon 带状截面；这会让尾流在某些视角下显得像一片扁带，而不是连续的体积等离子尾迹。

头顶名字与三角标记当前也是在 [GameScene.ts](D:/Projects/TypeScriptProjects/racing/src/scenes/GameScene.ts) 中用 `THREE.Sprite` 与 `THREE.Group` 放在 3D 世界里，只是通过 `depthTest: false` 保证穿透地图。这种方式天然会随透视缩小，导致远距离时名字明显变小，即使可见，也不符合“固定大小、稳定读法”的用户预期。

本次变更需要在不重写比赛玩法逻辑、不引入新依赖、不改动物理与性能公平规则的前提下，完成两项视觉/交互层级的结构升级：

- 尾流从 ribbon 主导改为粒子体积主导。
- 头顶标记从世界空间 sprite 改为屏幕空间 overlay。

## Goals / Non-Goals

**Goals:**

- 让尾流看起来更像连续的气态体积尾迹，而不是一条 ribbon 带。
- 让尾流的雾团结构、粒子感、流体感和前后密度变化更清晰，同时保持亮度不过曝。
- 保持尾流视觉长度与实际 draft 生效长度一致。
- 让尾流 draft 判定横向略宽于视觉本体，提高玩家跟尾流上手性。
- 让 NPC 名字与三角标记在屏幕上保持固定像素大小，不因距离变小。
- 让标记继续穿透地图稳定可见，并保留近距透明和屏幕中心遮挡抑制。
- 保持玩家自身不显示名字与三角。

**Non-Goals:**

- 不重新设计 draft charge 的积累/衰减机制。
- 不修改车辆基础性能、公平性、碰撞减速和动态 FOV 的既有玩法规则。
- 不引入 GPU 粒子框架、后处理依赖或外部体积贴图资源。
- 不将标记扩展为复杂雷达、箭头追踪器或屏幕边缘引导系统。
- 不把尾流改成真正流体模拟或 shader tube 管线。

## Decisions

### 1. 尾流视觉主结构改为“体积粒子簇”，不再以 ribbon mesh 为主要形态

当前方案虽然有三层 mesh，但用户已经多次明确“不需要 ribbon”。因此新的视觉主结构应直接转为“采样路径上的多团体积 sprite”，而不是继续保留 ribbon 再做弱化。

推荐结构：

- 每个有效 trail sample 生成 1 个主雾团
- 每个 sample 再生成 2 到 4 个偏移副团
- 每隔若干 sample 生成少量稀疏小粒子

这样视觉上会形成沿路径连续分布的体积云团链，而不是一条有明确边缘的带状面。

为什么不用继续保留 mesh：

- ribbon 在不同视角下容易变成不自然的片状带子
- 用户目标已经从“TRON 带状线”转向“立体气态尾流”
- 粒子簇结构更容易做出密度变化、流动感和体积感

备选方案：

- 保留 `trailShell / trailFin / trailCore` 但把透明度压低，只让 glow sprite 主导
- 这个方案改动较小，但会继续留下 ribbon 的几何痕迹，不符合当前方向，因此不推荐

### 2. 视觉宽度与实际 draft 判定宽度显式拆分

当前 `zoneRadius` 同时承担“玩家能不能吃到尾流”的感受责任，但视觉尾流又不一定要画得非常粗。为了兼顾易上手和美术控制，需要把这两个概念拆开：

- `visualRadius` / `visualClusterSpread`
- `draftZoneRadius`

约束建议：

- 视觉长度与实际判定长度保持一致
- draft 判定横向宽度可以比视觉宽度大 `1.25x ~ 1.45x`

这样玩家更容易“吃到”尾流，但不会被迫看到一条特别粗的蓝色带子。

备选方案：

- 继续强行让视觉宽度和判定宽度完全相同
- 这个方案更“物理直觉”，但手感更苛刻，也和用户“更好上手”的目标冲突

### 3. 保留现有 effectiveTrailPositions 路径裁剪，不重写长度生效逻辑

当前已经有 `effectiveTrailPositions`，并且用户之前专门要求“视觉长度和实际生效长度一致”。这一层逻辑已经符合目标，应继续作为唯一的路径真源。

新的视觉尾流只改变“怎么画”，不改变“哪一段有效”：

- 视觉团簇基于 `effectiveTrailPositions`
- draft 检测仍基于 `effectiveTrailPositions`
- 速度驱动长度增长仍基于当前 length curve

这能显著降低回归风险，也能避免再次出现“看起来很长但吃不到”的问题。

### 4. NPC 名字与三角改为屏幕空间 overlay，而不是 3D sprite 距离补偿

如果继续使用世界空间 sprite，再通过距离反向补偿 scale，可以近似实现固定大小，但仍会受到 FOV、透视和相机角度的影响，稳定性不够。

更适合当前需求的做法是：

- 保留每台 NPC 的 3D 头顶锚点
- 每帧把锚点 `project(camera)` 到屏幕坐标
- 在 HUD 层绘制固定像素尺寸的名字和三角

这样可以天然满足：

- 大小固定
- 远近一致
- 穿透地图
- 更易控制透明度和重叠抑制

备选方案：

- 保持 3D sprite，仅通过 scale 补偿距离
- 这个方案改动较小，但长期稳定性差，不推荐

### 5. 名字和三角继续作为一组 overlay 元素统一淡出

用户已经明确希望名字和三角是成套识别符号。因此无论是在近距淡出、屏幕中心遮挡抑制，还是潜在的重叠治理里，都应该把名字和三角当作一个整体处理，而不是分离控制。

建议规则：

- 固定字体大小
- 固定三角大小
- 仅透明度随规则变化
- 距离不再影响尺寸

### 6. 字体风格从霓虹艺术感转向更细的竞技 HUD 风格

当前世界 sprite 名字使用较粗字重和较明显的 glow，适合“艺术化场景字”，不适合稳定的比赛识别标记。

新的 overlay 字体方向建议：

- 字重 `400` 或 `500`
- 极细暗描边或无描边
- 很弱的外发光，甚至完全取消
- 固定字号，比如 `18px ~ 22px`

这样能让标记在远近上更均匀，不会显得“标题化”。

## Risks / Trade-offs

- [粒子数量变多导致每帧更新开销上升] → 控制每个 sample 的团簇数量，复用 sprite/material，对远端 sample 降密度
- [尾流失去 ribbon 后轮廓可能变得过散] → 保留“主雾团 + 副团 + 稀疏粒子”的三层密度结构，确保中心轮廓存在
- [视觉宽度和判定宽度分离可能引发“看着没碰上也吃到了”] → 保持倍率温和，仅做轻度放宽，不做夸张辅助
- [overlay 标记可能与 HUD 其他元素竞争层级] → 将 marker 容器独立成 race HUD 子层，明确 z-index 与 pointer-events 规则
- [固定大小标记在大量 NPC 聚集时更容易重叠] → 保留现有近距淡出与中心淡出，并为后续添加横向轻微避让或重叠压制留口

## Migration Plan

1. 在不动现有 draft 长度逻辑的前提下，先拆出新的尾流视觉参数与判定宽度参数。
2. 将尾流视觉从 mesh 主导切换为体积 sprite/particle 主导，同时保留现有 `effectiveTrailPositions`。
3. 在验证视觉长度与 draft 长度仍一致后，再调宽 draft 横向判定。
4. 引入屏幕空间 marker 容器和 3D 锚点投影逻辑。
5. 移除或停用当前世界空间 NPC marker sprite 方案。
6. 调整字体、字号、三角大小与透明度曲线，并在开局扎堆场景下验证。

回退策略：

- 若尾流视觉新方案不稳定，可保留采样与判定逻辑，仅回退到旧的视觉 renderer
- 若 overlay marker 出现严重 UI 层级问题，可短期回退到旧 marker 渲染路径，但保留参数和锚点抽象
