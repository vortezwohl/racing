## Context

当前比赛场景的载具逻辑集中在 `GameScene`、`Vehicle`、`Player` 和 `CPU`。`Vehicle` 已经创建透明 box hitbox，并将其用于赛道碰撞；但载具之间没有碰撞处理，尾流也不存在。`Player` 使用固定 FOV 的跟随相机，`CPU` 使用随机 `maxThrust`，各车辆数据文件也定义了不同的基础性能参数。

菜单中的车辆名称通过 canvas texture 生成 sprite，并挂载到车辆模型 group 下。由于每辆车的 `menuScaleMultiplier` 和模型缩放不同，名称会继承父级缩放，造成视觉大小不一致；同时 `Orbitron` 等字体没有在页面中显式加载，浏览器可能使用 fallback 字体。

本变更需要在不引入外部物理引擎、不重做赛道和模型的前提下，补齐身份识别、尾流、碰撞、动态 FOV 与公平性能规则。

## Goals / Non-Goals

**Goals:**

- 让玩家在比赛中清晰识别 NPC 位置、名称和身份颜色。
- 为所有载具提供一致长度规则、同色身份绑定的 TRON 风格尾流。
- 让尾流成为可吃到的非阻挡增益区域，支持持续累积和离开后衰减。
- 让载具之间存在实体体积，不再互相穿越。
- 让碰撞产生短暂减速和基于相对速度的挤压偏移。
- 让玩家相机 FOV 根据速度平滑变化，增强速度感。
- 统一所有载具基础性能，确保外观差异不带来性能差异。
- 修正菜单车辆名称字体和视觉尺寸不一致的问题。

**Non-Goals:**

- 不接入 Cannon、Ammo、Rapier 等外部物理引擎。
- 不实现联网同步或多人网络状态。
- 不重做 GLB 模型资产。
- 不改变赛道几何、检查点顺序或比赛完成规则。
- 不把尾流做成阻挡物；尾流没有 hitbox，只提供增益判定。

## Decisions

### 1. 使用统一 RaceVehicleRuntimeState 承载身份、尾流和增益状态

为每台参赛车辆建立运行时状态，而不是把所有新字段散落在 `GameScene` 中。状态至少包含：

- `id`
- `displayName`
- `identityColor`
- `markerGroup`
- `trail`
- `draftCharge`
- `collisionSlowUntil`

理由：身份标记、尾流和碰撞都需要按车辆遍历。集中状态可以降低跨模块副作用，也方便后续调试。

备选方案：直接在 `Vehicle` 上追加所有字段。该方案实现更快，但会让基础车辆类同时承担渲染身份、玩法状态和系统协调职责，长期可维护性较差。

### 2. 身份颜色在比赛开始时稳定分配

每个参赛者在 `GameScene.render` 创建后获得一个颜色。颜色从预定义高对比调色板中按顺序或可复现洗牌分配，不在每帧重新随机。

理由：玩家需要建立“颜色=载具身份”的稳定认知。完全随机虽然热闹，但可读性和测试稳定性更差。

备选方案：每局完全随机颜色。可作为后续增强，但第一版应避免颜色过近、对比不足和测试不可复现。

### 3. 玩家自己不显示头顶标记，NPC 始终显示

每台非本地玩家车辆显示名字 sprite 和同色三角指示器；本地玩家的 marker 不创建或保持不可见。

理由：玩家自己的标记会挡视野，且用户明确要求玩家看不到自己头顶名字和三角形。

备选方案：玩家也显示半透明标记。该方案违背需求，不采用。

### 4. 菜单名称与比赛名称使用同一套文本 sprite 工具

抽出或复用统一的 canvas text sprite 创建逻辑，确保字体、字号、字重、画布比例和缩放规则一致。菜单 label 不再直接继承车辆模型差异缩放，或者在挂载时做反向缩放抵消父 group scale。

理由：现有视觉不一致主要来自父级缩放和字体加载不确定性。修正根因比逐车调参更稳。

备选方案：为每个车型单独调 `labelY` 或 label scale。该方案会继续制造手工参数债，不采用。

### 5. 尾流使用采样点 ring buffer 加可视化 ribbon/sprite 段

每台车每帧记录尾部采样点，保存固定数量的历史点。视觉层根据速度决定可见采样长度，并使用同色透明材质、AdditiveBlending 和渐隐 alpha 绘制短尾流。

理由：车辆数量少，采样点方式简单可控，便于实现“速度越快尾流越长，但不超过上限”。相比真实流体模拟，它更稳定、成本更低，也更容易调试。

备选方案：用粒子系统或 shader tube。视觉潜力更高，但第一版复杂度更大，容易影响性能和可维护性。

### 6. 尾流增益使用距离线段检测，不使用物理 hitbox

尾流玩法判定使用“车辆位置到其他车辆尾流历史线段的距离”判断是否进入尾流区域。进入后累积 `draftCharge`，离开后按时间衰减。

理由：尾流没有 hitbox，且应可被吃到而不阻挡。线段距离检测比创建大量透明碰撞体更轻量，也更符合需求。

备选方案：为尾流创建 invisible mesh 并 raycast/boxcast。该方案调试复杂且容易误以为尾流有实体碰撞，不采用。

### 7. 载具碰撞第一版采用 AABB 粗碰撞加最小分离

每帧对参赛车辆做 pairwise 检测，使用 `Box3.setFromObject(vehicle.hitbox)` 判断重叠。发生重叠时按中心差向量分离双方，并根据相对速度分配挤压偏移和减速。

理由：当前车辆数量极少，O(n²) 足够。AABB 虽不如 OBB 精确，但实现和调试成本低，能先满足“不可随意穿越”的核心体验。

备选方案：OBB 或 SAT 精确碰撞。精度更好，但会显著增加实现复杂度；可作为后续优化。

### 8. 碰撞减速与尾流增益通过运行时倍率影响性能

保留统一基础性能，所有临时变化都通过运行时倍率计算：

- `collisionSlowScale`
- `draftAccelerationScale`
- `draftMaxSpeedScale`

理由：这样可以证明“基础性能一致”，同时允许尾流和碰撞作为比赛状态临时改变手感。

备选方案：直接改写车辆的 `acceleration` 或 `friction` 字段。该方案容易让临时状态泄漏到重置后，不采用。

### 9. 动态 FOV 只影响本地玩家相机，并且平滑插值

FOV 根据玩家速度占当前临时速度上限的比例计算，静止时较小，高速时变大，尾流增益可额外少量增加。每帧使用 lerp 平滑到目标值，并调用 `updateProjectionMatrix()`。

理由：FOV 跳变会眩晕，且 NPC 不需要相机状态。把逻辑限制在 `Player` 或本地相机更新路径中，影响面最小。

备选方案：同步改变相机距离和 FOV。第一版先只做 FOV，避免同时改变多个变量导致调参困难。

### 10. 公平性能通过单一配置源保证

新增统一基础性能配置，所有车辆运行时读取同一份参数。车辆数据仍可保留模型路径和体积。移除 CPU 随机 `maxThrust`，CPU 只能通过路线跟随、反应和转向策略表现差异，不得获得随机基础性能差异。

理由：用户明确要求所有载具性能完全一致。当前车辆数据和 CPU 随机值均违反此目标，必须收敛为单一来源。

备选方案：手动把每个车辆文件参数改成相同值。短期可行，但未来新增车辆容易再次漂移；统一配置更可靠。

## Risks / Trade-offs

- 尾流透明材质过多可能影响性能 -> 控制采样点数量、车辆数量和可见段数量，优先使用可复用 geometry/material。
- AABB 对旋转载具不够精确 -> 第一版以可用和稳定为先；若出现明显误判，再升级到 OBB。
- 动态 FOV 过大可能眩晕 -> 采用保守上限和平滑插值，并在 debug 模式暴露 FOV 参数便于调试。
- 头顶标记可能遮挡画面 -> 本地玩家隐藏标记，NPC 标记控制尺寸和高度；必要时按距离缩放。
- 尾流增益可能破坏公平 -> 增益必须来自同一规则，所有车辆都能吃尾流，且有上限和衰减。
- hitbox 可能穿帮可见 -> 正式模式材质必须透明、`opacity = 0`、不写深度；仅 debug 模式允许线框显示。

## Migration Plan

1. 新增统一运行时状态与公平性能配置，不改变现有比赛入口。
2. 修正菜单 label 缩放和字体加载，先验证现有菜单行为不退化。
3. 增加 NPC 头顶身份标记，玩家自身隐藏。
4. 增加尾流视觉但暂不启用增益，先验证视觉和性能。
5. 启用尾流增益状态，验证累积、上限和衰减。
6. 增加载具碰撞处理，验证不穿越、减速和挤压偏移。
7. 增加动态 FOV，验证低速和高速过渡。
8. 最后做整体 build 和手工比赛验证。

回退策略：每个切片都应保持独立，若某一项体验不稳定，可先关闭对应系统入口，不影响其他已完成切片。

## Open Questions

- NPC 名称是否沿用菜单车型名，还是使用独立赛车手名称，例如 `Nova-01`、`Comet-02`。
- 尾流增益数值需要最终手感调参，建议实现后用 debug 参数确认。
- 头顶标记是否需要被赛道遮挡；当前建议不遮挡，以满足“直接看到 NPC 位置”的体验。
