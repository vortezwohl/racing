## Context

当前竞速逻辑已经具备尾流积累、尾流离开后缓慢衰减、玩家镜头随速度变化和车辆体积碰撞等基础能力，但这几类反馈还没有完全区分清楚：尾流收益已经存在，却没有把“吃到尾流”塑造成足够明显的速度体验；碰撞也已经存在，却把前后撞和侧撞混为同一套减速 + debuff 行为。当前改动需要在不重写车辆物理与尾流体系的前提下，把参数收益和碰撞方向感调到更符合当前目标的状态。

## Goals / Non-Goals

**Goals:**
- 提高尾流状态下的 FOV 反馈，让玩家更容易感知自己正在吃尾流。
- 轻度提高尾流状态下的最高速度收益，同时保持现有加速度手感框架。
- 将车辆碰撞明确拆分为“侧撞”和“前后撞”两类行为。
- 保留侧撞现有 debuff 语义，但强化快车撞慢车的压制表现。
- 将前后撞改为回弹响应，取消原有减速和 debuff。

**Non-Goals:**
- 不改动尾流积累速率、离开尾流后的衰减机制、或尾流视觉渲染结构。
- 不重写车辆物理系统，不引入新的刚体或连续碰撞框架。
- 不调整 checkpoint、UI 标记、载具选择、或菜单系统。
- 不改变所有载具基础公平性，只在尾流附加收益和碰撞响应层调整手感。

## Decisions

### 1. 尾流收益只调数值，不改积累机制
尾流已经通过 `draftCharge` 驱动加速度、最高速度和 FOV。当前需求只需要增强收益表现，不需要新增第二条尾流状态链。因此最小且最稳的做法是：
- 把 `draftAccelerationBonus` 调整为 `0.5`
- 把 `draftMaxSpeedBonus` 调整为 `0.25`
- 把 `draftFovBonus` 调整为 `8`

这样可以保持现有尾流积累、衰减和公平性逻辑不变，只提高最终反馈强度。

备选方案：
- 重写尾流状态机，让 FOV 和速度额外依赖独立 charge
- 该方案可调性更高，但会引入新的状态耦合与验证成本，不适合当前最小目标

### 2. 碰撞在现有 `separateCollidingVehicles()` 内部分型，而不是拆出新碰撞系统
当前碰撞已经集中在 `GameScene.separateCollidingVehicles()`。要最小实现方向差异，最稳的方式是在这个函数内部先判定碰撞类型，再走不同分支，而不是把碰撞完全拆到新模块。

分型依据：
- 计算碰撞法线 `normal`
- 分别与双方前进方向做点积
- 若碰撞法线与前进方向更接近平行，则视为前后撞
- 若更接近垂直，则视为侧撞

这样可以在不改动现有碰撞入口和循环结构的前提下，直接分支处理减速、debuff 和回弹。

备选方案：
- 用上一帧相对位移轨迹而不是法线来判碰撞类型
- 该方案理论上更贴近动态运动方向，但当前系统没有单独缓存相对接触轨迹，增加复杂度不划算

### 3. 侧撞保留 debuff，但按快慢差放大慢车受击效果
当前侧撞语义已经符合“碰撞有惩罚”的基础方向，因此不需要推倒重来。新目标是让快车撞慢车时，慢车被横向挤飞更多、减速更多。

实现原则：
- 保留当前 `applyCollisionSlow()` 与减速倍率调用
- 在判定为侧撞时，依据两车速度差提高慢车位移份额
- 同时提高慢车的速度损失比例
- 快车仍然受影响，但明显小于慢车

这样既延续现有碰撞惩罚语义，又加强差速碰撞的竞技反馈。

### 4. 前后撞取消 debuff，改成前车回弹略强于后车
当前需求明确指出，正前方和正后方碰撞不应减速，而应表现为回弹。最终确认口径为：
- 双方不减速
- 双方不吃 debuff
- 前车回弹比后车强 `15%`

实现原则：
- 判定为前后撞后，不再走当前统一减速乘算
- 不调用 `applyCollisionSlow()`
- 沿碰撞法线给双方写入短促回弹速度
- 前车回弹系数为 `1.15`
- 后车回弹系数为 `1.0`

这样能避免前后顶撞过于“肉”，同时又不会让后车通过顶撞持续推动车队。

## Risks / Trade-offs

- [尾流 FOV 提高后镜头冲击感上升] → 将增益控制在 `8` 而不是更激进的 `10`，先增强感知再保留可控性。
- [侧撞增强后慢车可能被横向甩得过猛] → 让速度差放大量设上限，避免极端速度差导致失控位移。
- [前后撞取消减速后可能出现长时间顶推] → 用前车更强的回弹 `1.15x` 打断持续贴推。
- [仅用碰撞法线分型在边角接触时可能不够完美] → 先采用法线阈值做最小实现，若试玩仍有误判，再单独迭代判定策略。
