## Context

当前项目的玩家引擎声完全由 [src/objects/Player.ts](/D:/Projects/TypeScriptProjects/racing/src/objects/Player.ts) 中的 Web Audio 代码实时合成：创建一个 `OscillatorNode`，设置为三角波，并将频率按 `50 + velocity.length() * 100` 线性映射后直接输出到扬声器。这个实现足够轻量，但存在三个问题：

- 音色基础过硬，三角波直接输出会让低速和中速段偏尖、偏“蜂鸣器”。
- 音高变化过于线性，低速起步时就容易显得抬升过快，高速末段缺少继续上扬的兴奋感。
- 没有频谱层面的动态控制，油门和加速阶段听感差异主要只靠频率变化，缺少柔和电子音色应有的开合感。

本次设计聚焦玩家引擎声，不扩展到 NPC、资源型音效、背景音乐或新的音频管理系统。约束是保持改动集中在现有玩家音频链路附近，不引入第三方依赖，不破坏现有 `disposeAudio()` 生命周期。

## Goals / Non-Goals

**Goals:**

- 将玩家引擎声基础波形替换为正弦波，获得更圆润的基础音色。
- 基于玩家速度比建立非线性主频曲线，实现低速段增幅慢、高速段增幅快的听感。
- 增加一个带轻度谐振的低通滤波器，并让截止频率受玩家加速度强度驱动。
- 使用轻度非线性加速度曲线，让滤波器开启过程自然，保留“柔和电子”而不是夸张尖锐的音色。
- 为频率与滤波器参数使用时间常数平滑，减少台阶感和爆音风险。

**Non-Goals:**

- 不重构全局音频系统，不引入统一音量分组、静音开关或资源预加载机制。
- 不修改菜单选择音、倒计时音、出界音、单圈完成音、完赛音等资源型音效。
- 不为 NPC 增加单独引擎声，不新增背景音乐。
- 不引入复杂的多振荡器、包络发生器、失真、延迟或立体声调制效果。

## Decisions

### 1. 保持单振荡器方案，但将波形改为正弦波

采用单个 `OscillatorNode`，并将 `type` 从 `triangle` 改为 `sine`。这能以最小节点复杂度得到更柔和、圆润的基础听感，也能避免三角波在高频段的额外泛音让整体质感过硬。

备选方案：

- 保留三角波并仅靠低通削尖：实现最省，但基础音色仍偏硬，和“柔和电子”目标不完全一致。
- 使用双振荡器叠层：能做得更厚，但会扩大改动范围，也更容易引入相位和音量平衡问题，不符合当前最小闭环目标。

### 2. 用速度归一化后的指数曲线控制主频

主频不再直接使用世界速度线性换算，而是先计算 `speedRatio = clamp(speed / effectiveMaxSpeed, 0, 1)`，再通过指数曲线得到 `speedCurve = pow(speedRatio, exponent)`，最后插值到 `idleHz ~ maxHz` 区间。

推荐首版参数方向：

- `idleHz`: 约 `58`
- `maxHz`: 约 `205`
- `speedExponent`: 约 `2.05`

这样可以让低速段更稳，高速末段更有冲劲，且不会在起步瞬间听起来像转速猛窜。

备选方案：

- 线性曲线：实现简单，但正是当前问题来源。
- 更大的指数值：高段会更刺激，但容易偏离“柔和”风格，后续调参空间也更窄。
- Logistic/S 曲线：控制更细，但对当前需求属于过度设计。

### 3. 使用单个低通滤波器并固定轻度谐振

在振荡器和输出之间插入 `BiquadFilterNode`，类型为 `lowpass`。`Q` 使用固定轻度谐振值，例如 `2.4` 左右，使截止频率附近略有存在感，但不形成刺耳共振峰。

节点链路为：

`OscillatorNode -> BiquadFilterNode(lowpass) -> destination`

备选方案：

- 直接输出不加滤波：无法满足用户对截止频率和谐振的明确要求。
- 动态调 Q：可玩性更高，但会提升调参复杂度，且当前需求只要求“有一定谐振”，固定值更稳。

### 4. 用“正向加速度强度”驱动截止频率，并做轻度非线性映射

截止频率的控制目标是“初期增长慢，后期稍快，但不要明显偏离线性”。为此，使用当前帧与上一帧的速度差估算正向加速度：

- `currentSpeed = velocity.length()`
- `speedDelta = max(currentSpeed - previousSpeed, 0)`
- `accelStrength = clamp(speedDelta / dt / referenceAcceleration, 0, 1)`
- `cutoffCurve = pow(accelStrength, cutoffExponent)`

然后把 `cutoffCurve` 插值到 `minCutoffHz ~ maxCutoffHz` 区间。

推荐首版参数方向：

- `minCutoffHz`: 约 `260`
- `maxCutoffHz`: 约 `1180`
- `cutoffExponent`: 约 `1.18`

这里不直接使用油门键状态作为滤波依据，因为用户要求的是“截止频率和加速度非线性正相关”，真实速度变化比按键更贴合该语义，也能自然覆盖滑行、碰撞减速和贴地恢复等情况。

备选方案：

- 直接用 `throttle`：实现简单，但它表达的是输入意图，不是实际加速度。
- 用车辆理论加速度配置：稳定但偏抽象，不能真实反映速度变化、阻力和地形影响。

### 5. 使用 Web Audio 参数平滑而不是逐帧硬赋值

频率与滤波器截止频率都用 `setTargetAtTime()` 进行平滑过渡，避免每帧直接写值产生台阶噪声。建议为主频设置更短的时间常数，为截止频率设置略长的时间常数，使音高反应灵敏、音色开合稍微更柔顺。

推荐方向：

- 主频平滑时间常数：`0.05 ~ 0.07`
- 截止频率平滑时间常数：`0.07 ~ 0.09`

备选方案：

- `linearRampToValueAtTime()`：可用，但逐帧重设 ramp 更容易积累计划事件，`setTargetAtTime()` 更适合持续控制。
- 不做平滑：最小代码量，但音质风险过高。

### 6. 参数优先内聚在 Player 内，必要时再外提到配置

首版优先把少量音频参数常量放在 `Player` 附近，保证实现集中、diff 小、便于快速试听与调参。如果后续需要开放调参或不同载具音色，再考虑迁移到 `raceConfig`。

备选方案：

- 立即外提全部参数到配置文件：对长期维护更友好，但这次只有单一玩家引擎声，先保持局部闭环更稳妥。

## Risks / Trade-offs

- [风险] 浏览器自动播放策略可能导致 `AudioContext` 初始为 suspended 状态，改动后更容易让人误判为“引擎声失效”。
  → 缓解：保留现有上下文生命周期，实施时确认继续依赖已有用户交互触发链路，并在验证时覆盖首次进入比赛场景的情况。

- [风险] 用速度差估算加速度会受帧率抖动影响，导致截止频率轻微波动。
  → 缓解：对 `dt` 参与计算后的加速度值做归一化夹取，并使用 `setTargetAtTime()` 平滑。

- [风险] 低通截止频率和主频同时变化时，如果参数区间设置不当，可能出现整体过闷或高段过亮。
  → 缓解：先按“柔和电子”目标使用保守参数区间，并把参数集中定义，便于后续试听微调。

- [风险] 仅用正弦波会让音色过于纯净，部分玩家可能觉得“推力感”不如三角波直接。
  → 缓解：用轻度谐振和速度后段更快的主频曲线补回存在感；若后续仍偏弱，再讨论是否增加极轻微二次谐波，而不是首版就扩展结构。
