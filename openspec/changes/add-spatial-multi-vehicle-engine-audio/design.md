## Context

当前项目的实时合成引擎声已经从单一玩家三角波推进到更柔和的玩家专属合成链路，但这条链路仍然只存在于 [src/objects/Player.ts](/D:/Projects/TypeScriptProjects/racing/src/objects/Player.ts) 内部。比赛中的 NPC 没有连续引擎声，因此玩家无法仅凭听觉稳定感知附近车辆的存在、远近、前后方位和左右压迫关系。与此同时，玩家已经明确希望不同载具有波形差异，玩家车默认更有攻击性的锯齿波，NPC 则在数种基础波形中随机化。

这意味着引擎声不再只是“玩家手感增强”，而是跨 `Vehicle`、`Player`、`NPC`、`GameScene` 的系统级能力。设计上必须解决四个核心问题：

- 如何让所有载具共享一套可维护的实时引擎声链路，而不是继续把能力钉死在 `Player`。
- 如何在不引入第三方库的前提下，实现接近现实的距离衰减、前后方向性和左右声像。
- 如何让玩家与 NPC 拥有不同默认音色，并让 NPC 的波形随机化在每场比赛内稳定。
- 如何在支持 `pulse` 的同时保持实现闭环可控，因为 Web Audio 没有原生 `pulse` 波形。

## Goals / Non-Goals

**Goals:**

- 将实时合成引擎声提升为所有载具共享的能力，并统一管理初始化、更新与释放。
- 让玩家车保持主导存在感，其他载具略低一档，并随距离连续衰减。
- 让前方车辆更清晰、后方车辆略更轻且更闷，左右位置能通过声像感知。
- 将玩家默认基础波形改为 `sawtooth`。
- 让 NPC 在 `sine`、`triangle`、`sawtooth`、`pulse` 中随机选择基础波形，并在单场比赛内保持稳定。
- 为 `pulse` 提供项目内可复用的自定义波形构造方案。

**Non-Goals:**

- 不引入完整 3D 音频引擎或外部音频库。
- 不重构菜单、倒计时、完赛、单圈完成或出界等资源型音效。
- 不实现混响、遮挡反射、地形回声、多监听器或 HRTF 级别的精细空间化。
- 不为每辆车建立完全独立的复杂合成器架构，例如多振荡器堆叠、包络或调制矩阵。

## Decisions

### 1. 将引擎声能力下沉到 Vehicle 层

当前玩家引擎声在 `Player` 内部闭合，无法自然复用给 NPC。为满足“所有载具都能发声”，应把实时引擎声能力迁移到 `Vehicle` 级别，由 `Vehicle` 提供：

- 音频节点字段
- 默认初始化方法
- 每帧参数更新方法
- 空间化衰减方法
- 统一释放方法

`Player` 和 `NPC` 只负责提供音色配置差异，例如默认基础波形、基础音量和随机波形选择。

备选方案：

- 继续保留 `Player` 私有实现，再给 `NPC` 复制一份：实现最快，但维护成本和参数漂移风险太高。
- 在 `GameScene` 里集中管理所有车辆声音：集中度高，但会让场景层知道太多车辆音色细节，耦合过重。

### 2. 使用“共享 AudioContext + 每车独立节点链路”

为避免每辆车都新建 `AudioContext` 带来的资源浪费和浏览器策略风险，建议整场比赛共享一个 `AudioContext`，但每辆车拥有独立节点链路。单车链路建议保持轻量：

`Oscillator -> Harmonic(optional) -> Filter -> Gain -> StereoPanner -> Destination`

这样可以满足：

- 每辆车可独立控制主频和滤波
- 每辆车可独立设置总音量与方向衰减
- 每辆车可独立设置左右声像

备选方案：

- 每辆车一个 `AudioContext`：资源更重，也不利于统一控制。
- 完全只用 `PannerNode`：理论更接近 3D，但对当前项目是复杂度过高的首版。

### 3. 空间化首版采用“距离衰减 + 前后方向性 + 左右声像”

为了模仿现实规律且保持实现可控，首版不直接上完整 3D HRTF，而采用三段式空间化：

- 距离：用平滑平方衰减，例如 `1 / (1 + k * d^2)`
- 前后：基于玩家前向向量与“玩家到目标载具方向”的点积计算方向性衰减
- 左右：基于玩家右向量与目标方向的点积设置 `StereoPannerNode.pan`

方向性除了音量，还应带一点频谱变化：

- 前方车辆：保留更高的截止频率
- 后方车辆：略降低额外截止频率，让听感更闷一些

备选方案：

- 只做距离衰减：实现最小，但无法满足“根据前后位置衰减”的要求。
- 直接完整 `PannerNode` 3D：更接近标准，但对于当前场景和调参目标不够可控。

### 4. 玩家默认主波形改为 sawtooth，NPC 波形按比赛生成时随机

玩家主波形直接改为 `sawtooth`，以提升穿透感和推进感。NPC 在生成时从固定候选集中随机选择，并把结果保存在实例字段中，保证单场比赛内不抖动、不重抽。

推荐集合：

- `sine`
- `triangle`
- `sawtooth`
- `pulse`

其中玩家仍可保留轻微泛音层，但应避免和 `sawtooth` 自身的丰富谐波叠加过度，必要时可降低玩家额外泛音增益。

### 5. 使用 PeriodicWave 实现 pulse，而不是假设存在原生 pulse

Web Audio 的 `OscillatorNode.type` 不支持 `pulse`。因此 `pulse` 需要用 `PeriodicWave` 构造，或者用可控近似方案模拟。为了保持结果稳定，建议明确采用 `PeriodicWave`，并为其定义温和版本的谐波权重，避免听起来过于尖锐或刺耳。

备选方案：

- 用 `square` 代替 `pulse`：实现简单，但不等价，也不符合用户明确要求。
- 用双 saw 相减近似：可行，但调参与维护复杂度更高。

### 6. 分步实施，先稳定多载具与空间化，再细调 pulse

虽然提案需要一次性覆盖最终目标，但实现计划应明确先后顺序：

1. 抽取 `Vehicle` 级实时引擎声能力
2. 接入所有载具与空间化衰减
3. 切换玩家 `sawtooth`
4. 为 NPC 接入 `sine/triangle/sawtooth`
5. 最后补 `pulse` 的 `PeriodicWave`

这样可以在每一步都保持可听、可构建、可回退。

## Risks / Trade-offs

- [风险] 所有载具同时发声会让整体混音变脏，尤其是多辆高谐波波形叠加时。
  → 缓解：NPC 默认总音量低于玩家，并限制 NPC 波形集合与额外泛音强度；必要时给远处车辆更积极衰减。

- [风险] 共享 AudioContext 需要明确所有车辆生命周期，否则场景切换时更容易残留节点或引用。
  → 缓解：把初始化、更新、释放都收口到 `Vehicle` 级公共方法，并在 `GameScene.dispose()` 中统一遍历调用。

- [风险] sawtooth + 额外泛音层可能让玩家声音过亮或刺耳。
  → 缓解：为玩家和 NPC 分别设置不同的泛音增益默认值，并通过低通截止范围限制高频。

- [风险] pulse 的 `PeriodicWave` 如果权重设置激进，会成为混音中最突兀的声源。
  → 缓解：将 pulse 定义为“柔化脉冲”，优先压低高阶谐波权重，而不是追求传统硬脉冲质感。

- [风险] 基于玩家相机前向做空间化会与极端镜头状态产生少量偏差，例如倒车观察或自由镜头。
  → 缓解：首版先以玩家主行进朝向/相机朝向中更稳定的参考为准，并把自由镜头下的差异视为已知限制。
