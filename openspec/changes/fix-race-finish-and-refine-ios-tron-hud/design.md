## Context

当前比赛页已经完成了 Canvas HUD 的基础接入，但存在三个需要同时收敛的问题。第一，完赛面板的触发仍然依赖 `GameScene.update()` 在帧循环中读取 `player.laps` 的结果，这使“最后一圈已过线但结算面板未出现”成为可能。第二，`RaceHud` 当前仍带有较多描边框、面板框和偏粗的发光文字，整体风格更接近厚重赛车游戏 UI，而不是用户要求的简约 iOS 风格。第三，比赛内不同的返回菜单入口如果直接各自切回菜单，会破坏与菜单点击 `PLAY` 已有黑幕帘转场之间的一致性。

约束条件：
- 保持现有 `webpack + TypeScript + Three.js + canvas HUD` 架构，不引入新框架。
- 不改重写菜单系统，不重做车辆物理或 NPC 驾驶逻辑。
- 设置面板仍然必须保持非暂停式行为。
- 比赛返回菜单时的视觉过渡必须与菜单点击 `PLAY` 时的黑幕帘效果一致，不能出现直接闪切。
- 方案要尽量复用现有 `GameScene`、`RaceHud` 与 `AppShell` 结构，减少扩散修改。

## Goals / Non-Goals

**Goals:**

- 让玩家完成最后一圈后，结算面板稳定触发，不再依赖脆弱的额外帧时序。
- 为完赛逻辑增加幂等保护，避免重复结算、重复音效和重复状态写入。
- 将比赛 HUD 重构为更轻、更细、更小的简约现代风格，并保留少量 Tron 色彩。
- 将设置按钮改成无外框齿轮图标，并提供清晰但克制的按压反馈。
- 在设置弹窗中增加 `RESUME`，让用户可以直接关闭弹窗继续比赛。
- 将比赛内所有返回菜单入口统一到同一条黑幕帘转场链路，确保切回菜单时先渐黑、再切场、再渐显。

**Non-Goals:**

- 不重构菜单页视觉语言。
- 不重设计排行榜信息结构，只调整其触发可靠性与视觉表达。
- 不引入新的字体加载链路或第三方动画系统。
- 不修改 NPC 赛道策略来配合本次完赛修复。

## Decisions

### 1. 将“玩家过线事件”提升为显式完赛触发源，而不是继续只靠场景轮询 `laps`

当前完赛触发条件位于 `GameScene.update()`，通过 `if (this.player.laps > this.hudState.totalLaps)` 来决定是否展示结果面板。这种做法的问题是，`laps` 的变化来源于 `Vehicle.handleTrackCollision()` 内的 checkpoint 判定，而完赛展示发生在稍后的另一段流程里，中间存在时序窗口。

本次方案将把“玩家穿过起终点并完成最后一圈”视为显式事件：

- `Vehicle` 在确认玩家合法通过 checkpoint 1 且完成最后一圈时，向 `GameScene` 报告一个明确的完赛事件。
- `GameScene` 在收到该事件时立即执行 `handleRaceFinish()`，而不是再依赖下一帧读取 `laps`。
- 同时保留场景级幂等保护，确保即使同一帧重复触发也只会完成一次结算。

这样做的原因：

- 直接缩短“圈数变化”与“结算展示”之间的链路。
- 更容易区分“完成一圈”和“完成比赛”两类事件。
- 后续如果要为过线加特效或统计，也有明确的事件切点。

备选方案：

- 继续保留现有帧轮询，只增加更多保护判断。放弃，因为仍然无法消除最后一圈漏触发的根因。
- 在 `Track` 中集中判断完赛。放弃，因为 `Track` 不适合感知玩家身份和 UI 生命周期。

### 2. 完赛逻辑保持 `GameScene` 单点收口，并增加幂等保护

`handleRaceFinish()` 继续作为唯一的赛后入口，但会补充以下约束：

- 只允许从“玩家已合法完成最后一圈”的事件进入。
- 一旦 `finished === true`，后续重复调用直接返回。
- 音效停止、结果面板状态切换、排行榜冻结时间写入，都只在首次调用时执行。

这样可以保证：

- 不会因为多次碰撞判定或重复过线导致多次弹面板。
- 设置弹窗、排行榜与赛后 HUD 状态始终由同一条逻辑维护。

### 3. HUD 视觉从“面板框驱动”改为“排版驱动”

当前 `RaceHud` 的视觉主要由多个描边面板组成，视觉重心落在边框，而不是信息本身。本次会把顶部 HUD 改成排版优先：

- 左上只保留 `LAP` 和 `POS` 的轻量文字块，不再给每项包大框。
- 中上 `TIME` 保持居中，但缩小字号、减弱描边和辉光。
- 右上 `KM/H` 改为轻量文本布局，不再使用大号面板框。
- 颜色以白字为主，只保留浅蓝辉光和极细分隔，营造轻微 Tron 味道。

这样做的原因：

- 更贴近用户提出的 iPhone / iOS 式简约感。
- 视觉层级会从“边框很大”转成“文字信息清楚”。
- 仍然保留少量 Tron 色彩，不会与菜单页完全脱节。

备选方案：

- 仅调小现有面板框和字体。放弃，因为仍然保留了厚重的框架语言，本质风格不变。

### 4. 齿轮按钮采用无外框图标 + 瞬时缩放按压反馈

设置按钮不再使用额外外框面板，而是单独绘制简洁齿轮图标。点击时通过短时缩放和亮度变化反馈“被按下”的感觉：

- `RaceHudState` 增加按钮按压状态或按压时间戳。
- `GameScene` 在热区命中时先写入 pressed 状态，再切换设置弹窗显示。
- `RaceHud` 在一个短暂时间窗口内使用较小缩放值绘制齿轮。

这样做的原因：

- 满足“点击有反应”的交互要求。
- 不需要引入 CSS 或 DOM 动画，继续保持纯 Canvas 绘制。

备选方案：

- 用颜色闪烁替代按压缩放。放弃，因为触觉感较弱。

### 5. 设置弹窗增加 `RESUME`，并继续保持非暂停式会话

设置弹窗会扩展为三个操作：

- `RESUME`
- `RESTART RACE`
- `EXIT TO MENU`

其中 `RESUME` 只关闭弹窗，不改变比赛计时与仿真。比赛在弹窗打开期间仍继续运行，这一点与当前需求保持一致。

这样做的原因：

- 用户要求“点一下继续比赛”，而不是只能点空白处或再次点齿轮。
- 明确操作文案比隐式关闭更稳定，也更适合触屏环境。

### 6. 所有比赛返回菜单路径统一复用 `AppShell` 黑幕帘转场

菜单点击 `PLAY` 时，当前工程已经存在一套可感知的黑幕帘进场效果。此次不单独在 `GameScene` 或结果面板里各写一份返回菜单动画，而是把“从比赛回菜单”的动作收口到 `AppShell` 层统一处理：

- 设置弹窗 `EXIT TO MENU` 调用共享的返回菜单转场入口。
- 结算面板 `BACK` 调用同一个返回菜单转场入口。
- 未来若新增比赛内其它直接回菜单动作，也必须复用该入口，而不是各自直接切场景。

统一转场序列为：

- 先将幕帘渐变到黑屏。
- 在黑屏已遮住内容时切换到菜单视图。
- 再将黑幕渐显消退，露出菜单页。

这样做的原因：

- 可以保证不同入口的返回菜单体验一致，不会一个入口淡出、另一个入口直接闪切。
- 转场控制权集中在 `AppShell`，避免 `GameScene`、结果面板、菜单场景分别维护一段易分叉的状态机。
- 以后若需要微调时长、缓动或遮罩样式，只需改一处。

备选方案：

- 在设置弹窗和结果面板里分别各写一段返回菜单动画。放弃，因为很容易出现一个入口忘记同步，最终体验不一致。

### 7. 结果面板保留信息结构，但改为更轻的玻璃卡片表达

赛后结果面板仍保留：

- 最终名次
- 平均时速
- 总成绩时间
- 每圈耗时
- 排行榜
- `BACK / RETRY`

但其视觉会统一为更轻的半透明玻璃卡片：

- 降低边框感和赛博装饰纹理
- 减弱发光强度
- 使用更细、更现代的排版

这样可以在不改变结果信息密度的前提下，使视觉方向与新 HUD 一致。

## Risks / Trade-offs

- [把完赛触发从轮询改成事件后，可能影响现有圈数音效与分圈记录时序] → 让“完成一圈”和“完成比赛”共享同一条过线源，但拆分成两个显式分支，分别测试。
- [HUD 去框后，如果层级控制不好，信息可能在复杂赛道背景上不够清晰] → 保留极弱的浅蓝辉光与少量局部遮罩，不完全取消可读性保护。
- [齿轮按压反馈如果持续时间过长，会显得卡顿或误以为未点击成功] → 将 pressed 动画限制为短时瞬态，由时间戳驱动自动恢复。
- [新增 `RESUME` 可能与“点击弹窗外关闭”的现有行为产生重复] → 保留点击外部关闭，但把 `RESUME` 定义为显式主路径。
- [如果返回菜单转场继续分散在多个入口，后续极易出现某个路径直接切回菜单而绕过黑幕] → 将比赛返回菜单动作强制经由 `AppShell` 的单一转场入口，避免状态分叉。

## Migration Plan

1. 先重构完赛触发源，使最后一圈过线时能显式调用 `handleRaceFinish()`。
2. 为完赛链路补上幂等保护，并验证两圈完成后结果面板稳定出现。
3. 在 `RaceHud` 中替换顶栏绘制样式，去除厚重框体，切换为轻量排版。
4. 重绘齿轮按钮并加入 pressed 动画状态。
5. 扩展设置弹窗为 `RESUME / RESTART RACE / EXIT TO MENU` 三操作。
6. 将设置退出、结算返回等菜单回退入口收口到 `AppShell` 统一黑幕帘转场，并验证切场时机位于黑屏期间。
7. 统一结果面板视觉为轻玻璃卡片风格。
8. 回归验证赛中 HUD、设置交互、赛后结算、返回菜单转场与重开路径。

回滚策略：

- 如果完赛事件化改造引发回归，可先只保留场景级幂等保护和额外 finish 兜底判断。
- 如果新视觉方向不满意，可单独回退 `RaceHud` 绘制实现，而不影响完赛修复逻辑。
- 如果统一转场接入影响面超预期，可暂时保留共享 `AppShell` 接口，同时单独禁用新入口，避免回到多处复制动画的状态。

## Open Questions

- `RESUME` 是否需要成为设置面板中的默认高亮选项，还是仅与其它两个操作保持同级视觉？
- 赛后结果面板中是否仍保留当前全部文案标签，还是进一步简化为更少的标题和更紧凑的数值布局？
- 返回菜单黑幕帘的时长是否应与菜单 `PLAY` 完全一致，还是允许在复用同一风格的前提下做更短的回退版本？
