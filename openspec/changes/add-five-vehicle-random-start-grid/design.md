## Context

当前比赛初始化逻辑集中在 `GameScene.render()`。玩家总是从 `track.startPoint` / `startDirection` / `startRotation` 直接出生，而 NPC 只从 `speeders[0..2]` 中挑选，并通过 `z` 轴 `+4 / -4 / +4` 的简单偏移生成。这带来三个问题：

- 比赛总人数不会稳定为 5。
- 当玩家选择 `bike` 或 `mustang` 时，NPC 车型池和玩家车型池分离，阵容规则不一致。
- 现有 offset 会重复使用同一个出生位，无法保证开赛瞬间不重叠。

仓库里目前一共有 5 台菜单可选载具，且每台车都定义了 `width` / `length`。这为“固定总数 5、优先不重复、必要时允许重复”的规则 B 提供了直接基础。与此同时，`TrackData` 目前只有单一的 `startPoint`、`startDirection` 和 `startRotation`，还没有显式的发车网格数据结构，因此本次设计应尽量在不重做赛道数据的前提下完成最小闭环。

## Goals / Non-Goals

**Goals:**

- 让每场比赛稳定生成 5 名参赛者。
- 让 NPC 选车遵循“尽量不重复，除非不得不重复”的规则 B。
- 为 5 名参赛者生成固定的 `2 前 3 后` 交错起跑位。
- 让 5 个车位在每次比赛开始时被随机分配。
- 让起跑间距由载具体积与安全边距共同决定，确保开赛瞬间 hitbox 不重叠。
- 尽量将改动限制在比赛初始化阶段，减少对现有比赛进行中逻辑的侵入。

**Non-Goals:**

- 不改赛道曲线、检查点或完赛规则。
- 不改比赛中的实体碰撞算法，只保证开赛初始摆位不重叠。
- 不新增联网同步、多人逻辑或复杂 AI 选车偏好。
- 不为每张赛道手工配置专属起跑位文件，除非现有通用算法无法满足最小闭环。

## Decisions

### 1. 参赛名单统一从 `menuVehicles` 构建

比赛阵容不再拆成“玩家使用全量菜单车，NPC 只使用前三台 speeder”。玩家仍由当前 `speederIndex` 决定车型，NPC 则从 `menuVehicles` 中补足剩余 4 个席位。

理由：`menuVehicles` 才是当前项目里完整且可展示的可选载具源。直接基于它构建参赛名单，才能统一玩家和 NPC 的车型集合规则。

备选方案：保留 `speeders` 作为 NPC 专属池。该方案会继续制造玩家与 NPC 阵容规则不一致的问题，不采用。

### 2. 规则 B 采用“两阶段补位”策略

NPC 选车流程分两段：

1. 先从除玩家车型外的唯一车型中随机抽取，直到抽满或唯一车型耗尽。
2. 如果仍不足 4 名 NPC，再从完整车型池中继续随机补位，此时允许重复。

理由：这正好满足“尽可能不重复，除非不得不重复”。在当前仓库只有 5 台车时，通常不会发生重复；当未来玩家可选车数少于总参赛数时，规则仍然成立。

备选方案：完全禁止重复。该方案会把“固定 5 人”绑定到“必须至少有 5 台不同车”的前置条件上，扩展性较差。

### 3. 起跑网格由运行时几何计算，不改赛道资源结构

本次不在 `TrackData` 中新增一整套静态 grid 坐标数组，而是基于：

- `track.startPoint`
- `track.startDirection`
- 由 `startDirection` 计算出的横向向量
- 最大车辆 `width` / `length`
- 一个统一安全边距

在运行时生成 5 个槽位。

理由：当前所有赛道数据都只有单一起点定义。对现有数据结构做最小扩展，比为每张赛道手工维护 5 个出生点更符合当前闭环目标。

备选方案：给每张赛道增加 `startGridSlots`。这个方案更显式，但当前只有一张主赛道，先引入通用算法成本更低。

### 4. 槽位形状固定为 `2 前 3 后`，并以交错队形摆放

网格采用两排：

- 前排 2 个槽位
- 后排 3 个槽位

后排中心对齐前排整体宽度，并在横向上形成交错效果。纵向上，后排位于前排后方；横向上，3 个后排槽位分别落在“左、中、右”，前排 2 个槽位落在“左内、右内”。

理由：这能直接满足你的排布要求，同时比“单列蛇形”更容易根据宽度和长度推导安全间距。

备选方案：把 `2 前 3 后` 理解为“前 2 台、后 3 台但不要求交错”。该方案不符合你明确要求的交错排开，不采用。

### 5. 车位随机，网格不随机

每场比赛只随机“参赛者与槽位的映射关系”，不随机“槽位几何本身”。也就是说，5 个位置的几何关系恒定，但谁站在哪个位置每局都会重新洗牌。

理由：这样既满足“起步位置随机分配”，又能保持可预测的空间安全边界，便于验证不碰撞。

备选方案：同时随机横纵偏移。该方案会显著增加碰撞风险和验证复杂度，不适合作为第一版。

### 6. 安全间距按最大车体尺寸加边距统一计算

横向间距基于最大 `width`，纵向排距基于最大 `length`，再额外增加固定安全边距。所有槽位使用同一套全局间距，而不是因车型不同做逐车微调。

理由：当前起跑碰撞风险主要来自布局重叠，不来自赛中动态行为。使用“最大尺寸 + 边距”的保守算法可以一次性覆盖所有车型组合。

备选方案：按每台车单独算局部紧凑排布。该方案理论上更节省空间，但实现和验证都更复杂，不适合当前需求。

### 7. 起跑网格生成应保持对碰撞/身份系统透明

`GameScene` 中后续的 `raceVehicleStates`、排名、尾流和碰撞逻辑都继续依赖最终生成出的 `player` 与 `npcs` 数组，不需要感知槽位计算细节。新逻辑只负责在创建车辆前准备好：

- 参赛名单
- 5 个槽位
- 名单到槽位的随机映射

理由：这样改动边界最小，也最符合“只修改当前任务必需部分”的原则。

## Risks / Trade-offs

- [起点附近赛道空间不足] → 先使用保守间距并基于当前主赛道验证；若个别赛道未来空间不足，再补充赛道级覆盖参数。
- [未来车型尺寸差异继续变大] → 统一基于最大尺寸计算，优先保证不重叠，必要时后续再增加赛道专属微调。
- [随机映射让调试复现变难] → 保持随机来源集中，必要时可在 debug 模式下固定种子或打印车位映射。
- [当前 5 台车下几乎不会触发重复策略，导致规则 B 不易被肉眼覆盖] → 在规格和任务中明确要求验证“唯一车型不足时允许重复”的逻辑路径，避免实现时遗漏。
